# Bonus Module: Git and GitHub for journalists

## Midweek message

Hello everyone,

By now you should have watched the video, completed the readings, and started the exercise. If you've asked Claude Code to initialize a Git repository and make your first commit, you're on track.

A few things coming up in the forums:

**"I don't understand why I need Git if Claude Code is doing everything."** Fair question. Here's the scenario that answers it: you ask Claude Code to refactor a script, and the new version breaks something that used to work. Without Git, the old version is gone. With Git, you tell Claude Code to show you the history and roll back to the version before the refactor. Git is your undo button for entire projects, not just individual files.

**"What's the difference between saving and committing?"** Saving writes the current state of a file to disk. Committing takes a snapshot of your entire project at that moment and stores it permanently in the project's history. You save constantly. You commit when you've reached a meaningful checkpoint — a feature works, a bug is fixed, a document is finished. Think of saving as writing notes on a sticky pad, and committing as filing that sticky pad in a dated folder you can always go back to.

**"I'm confused about branches."** That's normal. Branches are the hardest concept in Git to grasp at first. The short version: a branch lets you try something without risking what already works. You tell Claude Code "create a new branch called experiment" and then make changes freely. If the experiment works, you merge it back. If it doesn't, you delete the branch and nothing was harmed. The main branch stays untouched the whole time.

Keep going. Git is one of those things that feels abstract until you need it, and then you're grateful you learned it.
