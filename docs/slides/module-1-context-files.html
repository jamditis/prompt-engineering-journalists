THE PROBLEM

Every new chat starts from zero

  • AI doesn't know your beat
  • AI doesn't know your writing style
  • AI doesn't know what you worked on last session
  • You re-explain everything. Every. Time.

WRITING GOOD CONTEXT

What changes the output

  • Key entities -- mayor name, city council members, department heads
  • Style rules that differ from defaults (AP style, Oxford comma, dateline format)
  • Terminology the AI would confuse ("resolution" vs "ordinance")
  • Project structure -- what files exist, how they relate

GENERIC -- DELETE THESE

"Be helpful and accurate"
"Always verify facts before publishing"
"Write clearly"

SPECIFIC -- KEEP THESE

"The city uses 'resolution' for binding votes and 'ordinance' for legislation"
"Mayor: Linda Chen (D), elected 2024"

CLAUDE.md and context files 6 / 16
QUALITY CHECK

The deletion test

Before saving a line, ask: "If I delete this, would the AI's output change for someone who already knows my beat?"
If not -- cut it.

FAILS THE TEST

  • x  "Verify facts before publishing"
  • x  "Use professional language"
  • x  "Be concise and clear"
  • These are defaults. They change nothing.

PASSES THE TEST

  • >  "City council has 9 members, 5 needed for quorum"
  • >  "Use 'unhoused' not 'homeless'"
  • >  "Budget year runs Jul 1 -- Jun 30"
  • These are specific. They shape every output.

GETTING STARTED

The /init command

  • Type /init inside a Claude Code session
  • Claude scans your project and generates a CLAUDE.md with sensible defaults
  • Review it, then customize for your beat
  • This is your starting point, not your finished product
The best context files are edited over time. Start with /init, then refine as you learn what the AI needs to know.
# In your terminal:
$ cd ~/city-hall-beat
$ claude

# Inside the Claude session:
/init

# Claude analyzes your project and
# creates a CLAUDE.md tailored to
# what it finds in the directory

MEMORY TYPES

Four categories of memory

USER

Who you are

Your role, expertise level, what kind of work you do. Persists so you don't re-introduce yourself.

FEEDBACK

Corrections and confirmations

What to repeat, what to avoid. Accumulated corrections compound over time. The AI learns your preferences.

PROJECT

Ongoing work

Deadlines, decisions, current state. Keeps track of where you left off between sessions.

REFERENCE

Where to find things

Pointers to external resources, file locations, URLs. So you don't re-explain your project structure.

VERSION CONTROL

Commit your context

  • CLAUDE.md is infrastructure, not a personal note
  • Commit it to Git
  • When you push to GitHub, your context goes with it
  • When a colleague clones the repo, your accumulated beat knowledge is already there
Shared context = shared standards. New team members and AI sessions start from your collective knowledge, not from zero.
# Your project directory
city-hall-beat/
  CLAUDE.md       # committed
  stories/
  data/
  sources.csv

# A colleague clones:
$ git clone repo-url
$ cd city-hall-beat
$ claude

# Claude already knows the beat.
# Mayor, terminology, style rules
# -- all loaded from CLAUDE.md

Extending the system

CUSTOM PATTERNS

Structured error logs

  • Create files like LESSONS.md to track patterns
  • When something goes wrong, capture the failure category
  • Write the rule that prevents it next time
  • Over time, this log becomes more valuable than individual corrections
Pattern, not anecdote. Don't log "Claude got the mayor's name wrong." Log "Always verify elected official names against the city website roster."
# LESSONS.md

## Terminology
Problem: AI used "homeless" in
  draft copy.
Rule: Always use "unhoused" per
  style guide.

## Data sourcing
Problem: AI cited 2022 census
  data as current.
Rule: Flag any statistic older
  than 12 months. Prefer ACS
  1-year estimates when available.

## Attribution
Problem: AI paraphrased a direct
  quote without marking it.
Rule: Never alter quoted text.
  Use [paraphrase] label.
